## 一、整体架构

### 1.1 拆分范围

本次拆分的范围：**SDLC_Event（event-client + event-server）+ SDLC_Jira（独立服务）**。SDLC_Admin 和 SDLC_Export 保持 jar 不变，SDLC_POC 保持现状作为临时网关。

### 1.2 目标结构

```
SDLC/ (根 POM，统一管理版本)
│
├── SDLC_Common (jar)              ← 最底层：工具类、基础配置、公共常量、异常定义
│
├── SDLC_Admin (jar)               ← 登录+权限，保持 jar
│   ├── 被 poc-app、event-server、jira-server 依赖
│   └── 有独立 Controller（登录接口）
│
├── SDLC_Export (jar)              ← 导出功能，保持 jar
│   └── 被 poc-app 依赖
│
├── SDLC_Event (父 POM)            ← packaging: pom，只做聚合
│   ├── event-client (jar)         ← Feign 接口 + 共享 DTO/注解/Entity/Enum
│   │   └── 被 poc-app、jira-server 引用
│   └── event-server (bootable)    ← 启动类 + REST Controller + Validator/Handler 实现
│       └── 端口 :8081
│
├── SDLC_Jira (bootable)           ← 单模块，独立服务
│   │   ├── api/                   ← Feign 接口 + DTO（给调用方引用）
│   │   ├── service/               ← JiraGateway 实现 + 同步逻辑
│   │   └── controller/            ← REST Controller
│   └── 端口 :8082
│
└── SDLC_POC (bootable)            ← 主业务 + 临时网关
    └── 端口 :8080
```

### 1.3 运行时拓扑

```
Browser
   │
   ▼
┌─────────────┐      Feign       ┌─────────────────┐
│ SDLC_POC    │ ───────────────→ │ SDLC_Event      │
│ (:8080)     │                  │ event-server     │
│ 主业务+网关  │                  │ (:8081)          │
└─────┬───────┘                  └─────────────────┘
      │
      │ Feign
      ▼
┌─────────────┐      HTTP        ┌─────────────────┐
│ SDLC_Jira   │ ───────────────→ │ Jira Cloud API  │
│ (:8082)     │                  │ (外部)           │
└─────────────┘                  └─────────────────┘
      ▲
      │ 共用
      ▼
┌─────────────┐
│ Oracle DB   │
│ (同一个实例) │
└─────────────┘
```

### 1.4 关键依赖规则

- POC 依赖 event-client（通过 Feign 调 event-server），不直接依赖 event-server
- POC 依赖 SDLC_Jira 模块中的 api 包（通过 Feign 调 jira-server），不依赖其内部实现
- event-server 和 jira-server 都不依赖 POC
- Common 被所有模块依赖，Common 不依赖任何人

## 二、POM 改造 — 根 POM

### 2.1 根 POM 的定位

根 POM 不做 Spring Boot 的 parent，只做**版本统一管理**。各 bootable 模块（poc-app、event-server、jira-server）各自通过 dependencyManagement import 的方式拿到 spring-boot-starter-parent 的能力。

为什么？SDLC_Event 改造后下面有两个子模块，一个需要 spring-boot-maven-plugin 打 fat jar，一个不需要。如果根 POM 继承了 Boot Parent，所有模块都被影响。让需要 Boot 能力的模块自己处理，更干净。

### 2.3 Maven 双 parent 问题的解决

bootable 模块（event-server、SDLC_Jira、SDLC_POC）既要继承根 POM（版本管理），又要拿到 spring-boot-starter-parent 的能力（fat jar 打包、默认配置）。Maven 不支持多继承，解决方案是在子模块的 dependencyManagement 中用 scope=import 引入 Boot Parent：

```xml
<!-- event-server/pom.xml -->
<parent>
    <groupId>com.sdlc</groupId>
    <artifactId>SDLC</artifactId>
    <version>1.0.0</version>
</parent>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-parent</artifactId>
            <version>3.2.5</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

这样既继承了根 POM 的版本管理，又拿到了 Boot Parent 的所有默认配置和插件。

### 2.4 你需要做的事

1. 打开现有根目录的 pom.xml
2. 确认 groupId、artifactId、version 和当前项目一致
3. 把 properties 里的版本号对齐到你实际用的版本
4. 把 dependencyManagement 里缺的第三方依赖补上
5. 把 modules 对齐到实际的目录名
6. 执行 `mvn clean compile -pl .` 验证根 POM 本身能通过

## 三、SDLC_Event 拆分 — event-client + event-server

### 3.1 改造前的现状

```
SDLC_Event_Core/   → REST Controller + Service（门面）
SDLC_Event/        → Validator/Handler 实现（引擎）
```

Core 依赖 Event。改造后的 client/server 模式：

```
SDLC_Event/（父 pom）
├── event-client/  → 纯接口 + DTO + 注解（谁要调 Event 就引用这个 jar）
└── event-server/  → 启动类 + Controller + Service + Validator/Handler 实现（:8081）
```

**EventController — 直接 implements FeignClient，签名强绑定：**

```java
@RestController
@RequestMapping("/api/event")
@RequiredArgsConstructor
public class EventController implements EventFeignClient {

    private final EventExecutionService executionService;

    @Override
    @PostMapping("/validate")
    public List<ValidationResult> validate(@RequestBody ValidationRequest request) {
        return executionService.validate(request);
    }

    @Override
    @PostMapping("/execute")
    public ExecutionResult execute(@RequestBody ExecutionRequest request) {
        return executionService.execute(request);
    }

    @Override
    @GetMapping("/health")
    public String health() {
        return "OK";
    }
}
```

EventController 直接 implements EventFeignClient，这样控制器签名和 Feign 接口强绑定，改接口时两边一起报编译错，不会遗漏。

## 七、RequestContext — 跨服务上下文传递

### 7.1 问题场景

拆成三个独立服务后，一个请求可能经过多个服务。每个服务都需要知道"谁在操作""哪个租户""关联哪个 Jira Instance"。现在这些信息散落在方法参数里，层层传递。

### 7.2 设计目标

任何地方都能无感获取上下文信息，不需要 Controller → Service → DAO 层层传参。

### 7.3 实现机制：Filter + ThreadLocal + Feign RequestInterceptor

三步走：Filter 解析请求并设置上下文 → ThreadLocal 存储 → Feign RequestInterceptor 自动传播。

### 7.4 RequestContext（放在 SDLC_Common）

```java
package com.sdlc.common.context;

import lombok.Data;
import java.util.HashMap;
import java.util.Map;

/**
 * 请求上下文 — 一次 HTTP 请求的生命周期内共享
 *
 * 存储位置：ThreadLocal（线程绑定的一个 Map 里）
 * 生命周期：请求进入 Filter 时创建，响应返回后销毁
 *
 * 使用 Map 而不是固定字段，因为字段会随业务扩展而增加
 */
@Data
public class RequestContext {
    private String userId;
    private String username;
    private String accessToken;
    private String instanceId;
    private String cloudId;
    private String siteUrl;
    private String tenantId;
    private String traceId;
    private String requestId;
    private Map<String, String> extras;
}
```

### 7.5 Current 工具类（放在 SDLC_Common）

```java
package com.sdlc.common.context;

/**
 * 静态工具类 — 任何地方直接调
 *
 * 用法：
 *   String userId = Current.userId();
 *   String token = Current.accessToken();
 *
 * 不需要注入，不需要传参
 */
public class Current {

    private static final ThreadLocal<RequestContext> CONTEXT = new ThreadLocal<>();

    public static void set(RequestContext context) { CONTEXT.set(context); }
    public static void clear() { CONTEXT.remove(); }

    public static RequestContext context() { return CONTEXT.get(); }

    public static String userId() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getUserId() : null;
    }
    public static String username() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getUsername() : null;
    }
    public static String accessToken() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getAccessToken() : null;
    }
    public static String instanceId() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getInstanceId() : null;
    }
    public static String tenantId() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getTenantId() : null;
    }
    public static String traceId() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getTraceId() : null;
    }
    public static String requestId() {
        RequestContext ctx = CONTEXT.get();
        return ctx != null ? ctx.getRequestId() : null;
    }
}
```

### 7.6 ContextFilter（每个服务各自配置）

```java
@Component
@Order(1)
public class ContextFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;

        RequestContext context = new RequestContext();
        context.setUserId(httpRequest.getHeader("X-User-Id"));
        context.setUsername(httpRequest.getHeader("X-Username"));
        context.setAccessToken(httpRequest.getHeader("Authorization"));
        context.setTenantId(httpRequest.getHeader("X-Tenant-Id"));
        context.setInstanceId(httpRequest.getHeader("X-Instance-Id"));
        context.setCloudId(httpRequest.getHeader("X-Cloud-Id"));
        context.setSiteUrl(httpRequest.getHeader("X-Site-Url"));

        String traceId = httpRequest.getHeader("X-Trace-Id");
        context.setTraceId(traceId != null ? traceId : UUID.randomUUID().toString());

        String requestId = httpRequest.getHeader("X-Request-Id");
        context.setRequestId(requestId != null ? requestId : UUID.randomUUID().toString());

        try {
            Current.set(context);
            chain.doFilter(request, response);
        } finally {
            Current.clear();  // 必须！否则内存泄漏
        }
    }
}
```

### 7.7 Feign RequestInterceptor — 自动传播上下文（放在 SDLC_Common）

```java
@Configuration
public class ContextFeignInterceptor implements RequestInterceptor {

    @Override
    public void apply(RequestTemplate template) {
        addIfNotNull(template, "X-User-Id", Current.userId());
        addIfNotNull(template, "X-Username", Current.username());
        addIfNotNull(template, "Authorization", Current.accessToken());
        addIfNotNull(template, "X-Tenant-Id", Current.tenantId());
        addIfNotNull(template, "X-Instance-Id", Current.instanceId());
        addIfNotNull(template, "X-Cloud-Id", Current.cloudId());
        addIfNotNull(template, "X-Site-Url", Current.siteUrl());
        addIfNotNull(template, "X-Trace-Id", Current.traceId());
        addIfNotNull(template, "X-Request-Id", Current.requestId());
    }

    private void addIfNotNull(RequestTemplate template, String name, String value) {
        if (value != null) {
            template.header(name, value);
        }
    }
}
```

### 7.8 使用效果

```java
// 之前：层层传参
public void createIssue(IssueDTO dto, String userId, String token,
                         Long tenantId, String instanceId) {
    log.info("User {} creates issue", userId);
    issueDao.insert(dto, userId);
    notificationService.notify(userId, tenantId);
}

// 之后：零参数传递
public void createIssue(IssueDTO dto) {
    log.info("User {} creates issue", Current.userId());
    issueDao.insert(dto);           // DAO 内部自己调 Current.userId()
    notificationService.notify();   // 内部自己调 Current.tenantId()
}
```

### 7.9 文件归属

| 文件 | 放哪 | 原因 |
|------|------|------|
| RequestContext.java | SDLC_Common | 纯数据类，各模块都要用 |
| Current.java | SDLC_Common | 静态工具类，各模块都要用 |
| ContextFeignInterceptor.java | SDLC_Common | Feign 公共配置，各服务复用 |
| ContextFilter.java | 各自服务 | 每个服务从 JWT/Header 解析逻辑可能不同 |

### 7.10 注意事项

每次请求结束时必须 `Current.clear()`。Filter 的 finally 块已经保证了。但如果代码里手动开了新线程（比如 @Async），需要在新线程开头重新设置上下文，因为 ThreadLocal 不跨线程。

### 8.5 Projection 聚合表 — issue_detail

**两套方案对比：**

| 操作 | Oracle (CLOB) | PostgreSQL (JSONB) |
|------|--------------|-------------------|
| 写入 | detail_json = '{...}' 字符串拼接 | detail = '{"summary":"..."}'::jsonb 或 TypeHandler |
| 单字段更新 | 读出 → 反序列化 → 修改 → 序列化 → 写回 | UPDATE SET detail = jsonb_set(detail, '{status}', '"Done"') |
| 查嵌套字段 | 不能直接查 | SELECT detail->>'status' FROM issue_detail WHERE detail @> '{"priority":"High"}' |
| 索引 | 不支持内部字段索引 | GIN 索引，所有嵌套字段可查 |

### 8.7 Projection 更新机制

```
业务操作（如新增评论）
    │
    ▼
CommentService.addComment()
    │
    ├── comment 表插入数据
    │
    └── 发布 IssueChangedEvent
            │
            ▼
        ProjectionWorker（异步，事务提交后执行）
            │
            ├── 查询 issue + comment + attachment + history + worklog + custom_field_value
            ├── 组装 detail_json
            ├── INSERT/MERGE INTO issue_detail 或 INSERT ON CONFLICT UPDATE
            └── 更新 issue_detail.version = issue.version
```

为什么不用数据库 Trigger？Trigger 是 PL/SQL 代码，不好维护和测试；复杂 JSON 拼接性能差；未来拆微服务时跨库无法工作。Java 事件可以随时替换通道（现在直接调用，以后走 MQ）。

### 8.9 MyBatis JSONB TypeHandler（PostgreSQL 专用）

```java
package com.sdlc.common.handler;

import com.fasterxml.jackson.databind.ObjectMapper;
import org.apache.ibatis.type.BaseTypeHandler;
import org.apache.ibatis.type.JdbcType;
import org.postgresql.util.PGobject;

import java.sql.*;

public class JsonbTypeHandler extends BaseTypeHandler<String> {

    private static final ObjectMapper MAPPER = new ObjectMapper();

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i,
                                     String parameter, JdbcType jdbcType) throws SQLException {
        PGobject pgObject = new PGobject();
        pgObject.setType("jsonb");
        pgObject.setValue(parameter);
        ps.setObject(i, pgObject);
    }

    @Override
    public String getNullableResult(ResultSet rs, String columnName) throws SQLException {
        return rs.getString(columnName);
    }

    @Override
    public String getNullableResult(ResultSet rs, int columnIndex) throws SQLException {
        return rs.getString(columnIndex);
    }

    @Override
    public String getNullableResult(CallableStatement cs, int columnIndex) throws SQLException {
        return cs.getString(columnIndex);
    }
}
```

### 9.1 当前情况

有两套相关资产：jira-sync-engine（独立原型，DDD 四层架构）和 SDLC_Jira（SDLC 项目中的 Jira 模块）。目标是把 jira-sync-engine 原型的核心能力整合进 SDLC_Jira，让它从"直接调 Jira API 的封装层"升级为"带最终一致性保障的同步引擎"。

| Infrastructure：MyBatis-Plus Mapper | 新建 | 用 SDLC 项目的持久化规范 |
| Infrastructure：JdbcPollingMonitor | 保留 | 数据库变更采集 |

### 9.5 同步流程（两条路径）

**路径 A：即时同步（直接模式，当前）**

```
POC → Feign → SDLC_Jira
                  │
                  ├── JiraController.createIssue()
                  ├── DirectJiraGateway.createIssue()  → Jira Cloud API
                  └── 返回结果给 POC
```

适合低频操作。问题：Jira API 挂了整个请求失败。

**路径 B：最终一致性同步（新增）**

```
POC 修改 Issue
    │
    ├── issue.version = issue.version + 1
    ├── issue.dirty = TRUE
    └── 发布 IssueChangedEvent
            │
            ▼
      SyncScheduler（每 30 秒扫描）
            │
            ├── SELECT * FROM issue WHERE dirty = TRUE LIMIT 200
            ├── 对每个 Issue 收集变更 → 生成 SyncTask
            ├── 写入 Redis 幂等锁
            ├── 写入 sync_task 表（status = PENDING）
            │
            ▼
      TaskConsumer（每 30 秒消费）
            │
            ├── 拿 PENDING 任务
            ├── 双重幂等检查（Redis 锁 + sync_log 唯一索引）
            ├── 调用 JiraGateway.updateIssue() → Jira Cloud API
            ├── 记录 sync_log 审计日志
            ├── 成功后：issue.synced_version = issue.version, issue.dirty = FALSE
            └── 失败后：retry_count + 1, 计算 next_retry_time
```

### 9.6 SyncScheduler 逻辑

为什么 30 秒而不是实时？用户可能一秒改 10 次 — 不需要同步 10 次；30 秒的窗口自然聚合了高频修改；Jira API 有频率限制，批量比实时安全。

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class SyncScheduler {

    private final IssueMapper issueMapper;
    private final AggregationEngine aggregationEngine;
    private final SyncRedisService syncRedisService;
    private final SyncTaskMapper syncTaskMapper;

    @Scheduled(fixedDelay = 30_000)
    public void scanDirtyIssues() {
        List<IssueEntity> dirtyIssues = issueMapper.findDirtyIssues(200);

        for (IssueEntity issue : dirtyIssues) {
            try {
                issueMapper.markSyncing(issue.getId());

                List<ChangeEvent> events = collectChanges(issue);

                SyncTask task = aggregationEngine.aggregate(
                    events, issue.getIssueKey(),
                    issue.getUpdatedAt(), LocalDateTime.now()
                );

                if (task != null) {
                    if (syncRedisService.tryAcquireDedupLock(task.getIdempotencyKey())) {
                        syncTaskMapper.insert(toEntity(task));
                    }
                }
            } catch (Exception e) {
                log.error("Failed for issue {}: {}", issue.getIssueKey(), e.getMessage());
                issueMapper.markIdle(issue.getId());
            }
        }
    }
}
```

### 9.7 为什么需要 sync_task 和 sync_log 两张表

sync_task 是调度视角：什么东西还没同步、要不要重试、下次重试什么时候。sync_log 是审计视角：每次操作是否成功、错误信息、请求和响应。分开的好处是运维查失败原因看 sync_log，调度器决定同步内容看 sync_task，两者独立索引互不影响。

### 10.5 Automation Center — 定位

### 10.9 Automation Engine 核心逻辑

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class AutomationEngine {

    private final Map<String, AutomationAction> actionMap;
    private final AutomationRuleMapper ruleMapper;
    private final AutomationExecutionMapper executionMapper;

    @Scheduled(fixedDelay = 1000)
    public void scanRules() {
        List<AutomationRuleEntity> rules = ruleMapper.findEnabled();

        for (AutomationRuleEntity rule : rules) {
            if (!shouldExecute(rule)) continue;

            AutomationAction action = actionMap.get(rule.getActionClass());
            if (action == null) {
                log.warn("Action not found: {}", rule.getActionClass());
                continue;
            }

            long start = System.currentTimeMillis();
            AutomationExecutionEntity exec = new AutomationExecutionEntity();
            exec.setRuleId(rule.getId());
            exec.setRuleName(rule.getName());
            exec.setTriggerType(rule.getTriggerType());
            exec.setStartTime(LocalDateTime.now());

            try {
                action.execute(buildContext(rule));
                exec.setSuccess(true);
                exec.setDurationMs(System.currentTimeMillis() - start);
            } catch (Exception e) {
                exec.setSuccess(false);
                exec.setErrorMessage(e.getMessage());
                exec.setDurationMs(System.currentTimeMillis() - start);
                log.error("Automation rule '{}' failed", rule.getName(), e);
            }

            exec.setEndTime(LocalDateTime.now());
            executionMapper.insert(exec);
        }
    }

    private boolean shouldExecute(AutomationRuleEntity rule) {
        return switch (rule.getTriggerType()) {
            case "CRON" -> CronUtil.match(rule.getCronExpression(), LocalDateTime.now());
            case "MANUAL", "API" -> true;
            default -> false;
        };
    }
}
```
