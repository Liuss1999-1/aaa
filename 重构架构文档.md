## 1. 约束条件与核心决策

### 1.1 硬约束

| 约束 | 详情 |
|------|------|
| **时间** | 仅 3 天，1 人专注重构，另 1 后端继续开发功能 |
| **系统状态** | 未上线，功能不完整，上线后目标 ≈ 万级用户 |
| **数据库** | Oracle 当前在用，PostgreSQL 延后迁移 |
| **缓存** | 当前内存模拟，需切换真实 Redis |
| **团队规模** | 2 前端 + 2 后端 + 1 测试 + 1 运维 |

### 1.2 核心决策

| 决策 | 结论 | 原因 |
|------|------|------|
| 3 天内是否拆微服务？ | **是，拆 Event 为独立服务** | 循环依赖 + 规则引擎天然适合独立；3 天够做 |
| 是否迁移数据库？ | **否** | Oracle → PostgreSQL 放后续阶段；当前共用同一 DB |
| Redis 何时引入？ | **Day 2** | 从内存模拟切换到真实 Redis，独立为 Common 级模块 |
| 微服务通信方式？ | **OpenFeign** | 无 Kafka/RabbitMQ，同步调用即可；共用 DB 暂避分布式事务 |
| 服务发现方式？ | **当前硬编码 / 环境变量；未来 AWS ECS Service Discovery** | 不做过度设计，上线前 1-2 周引入 DNS 发现即可 |

### 1.3 什么是「上线后再改就难了」？

这是贯穿 3 天所有决策的判断标准：

- **循环依赖** → 不改，未来任何改动都互相锁死
- **模块边界** → 上线后改边界 = 需要协调多人改 import
- **CI/CD** → 没有自动化，上线后代码质量不可控
- **Jira 扩展点** → 直接调 API 的代码散落各处，未来改同步模式需全量翻改

## 2. 整体重构路线

### 2.1 路线图

```
┌──────────────────────────────────────────────────────────────────────────────────┐
│                              现在 ───────────────────────────────→ 未来           │
│                                                                                  │
│  ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌──────────────┐    ┌───────────┐ │
│  │ Phase 0 │───→│ Phase 1 │───→│ Phase 2 │───→│   Phase 3    │───→│ Phase 4   │ │
│  │ 现状    │    │ 模块重组│    │ 功能完善│    │ 微服务拆分   │    │ 云原生化  │ │
│  │ (已)    │    │ (3天)   │    │ (2-3月) │    │ (1-2月)      │    │ (持续)    │ │
│  └─────────┘    └─────────┘    └─────────┘    └──────────────┘    └───────────┘ │
│                                                                                  │
│  单体项目        Event拆微服务    稳定功能        拆 Export/Admin    数据库独立    │
│  循环依赖        解循环依赖       Redis缓存       独立数据库        PostgreSQL    │
│  内存缓存        Redis模块化      CI/CD运转      服务发现          AWS 部署      │
│  无 CI/CD        Jira扩展点       POC内包整洁    监控体系          Secret管理    │
│                  CI/CD就绪                                                        │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 各阶段定位

**Phase 0（当前）**：知道问题，未开始改。

**Phase 1（3 天 — 本次执行）**：专做「上线后改不了」的结构性手术。不增加新功能，不迁移数据库。

**Phase 2（功能完善期，2-3 月）**：在已干净的架构上继续开发功能。CI/CD 运转、代码审查按 ArchUnit 规矩来。技术债不再新增。

**Phase 3（微服务拆分期，1-2 月）**：功能稳定后，按已定义的服务边界拆 Export、Admin。各服务独立数据库。此时架构已支撑拆分——只需加 Feign + 独立部署，不需要重构。

**Phase 4（云原生化，持续）**：AWS ECS / K8s 部署、Secrets Manager、监控、日志、灰度发布。

## 3. 模块重组方案

### 3.1 当前结构（问题版）

```
SDLC_Common
├→ SDLC_Admin
│    ├→ SDLC_Export
│    └→ SDLC_Event ←────┐  (循环依赖, @Lazy)
└→ SDLC_POC ────────────┘
     ├── 11 个 Controller
     ├── 全部 Service
     ├── 全部 DAO
     └── 全部 Entity
```

**问题清单**：
1. Event ↔ POC 循环依赖（5 处，@Lazy 掩盖）
2. POC 巨型模块承载全部业务
3. Jira 集成散落在各个 Service 中
4. 无 Redis 模块，内存模拟不可靠
5. 无 CI/CD

### 3.2 目标结构（Phase 1 结束后）

```
SDLC_Common                  ← 最底层：工具类、基础配置、公共常量
│
├→ SDLC_Redis                ← 缓存模块（无上游依赖），jar
│
├→ SDLC_Admin                ← 管理模块，jar
│    ├→ SDLC_Export          ← 导出模块，jar
│    └→ SDLC_Event (父)      ← packaging: pom
│         ├── event-client   ← jar：Feign 接口、DTO、Entity、注解、Validator/Handler 接口
│         └── event-server   ← bootable (:8081)：启动类、REST Controller、Validator/Handler 实现
│
└→ SDLC_POC (父)             ← packaging: pom
     ├── poc-app             ← bootable (:8080)：主业务（Issue、Tollgate、Attachment、Tailoring 等，Jira 交互已剥离）
     ├── poc-jira            ← bootable (:8082)：Jira 集成服务（封装 JiraGateway，独立部署，重启不影响主应用）
     └── 各子模块依赖 event-client（通过 Feign 调用 event-server）
```

**依赖方向**：全部自上而下，零回环。

```
Common ← Redis
Common ← Admin
Common ← Event (event-client, event-server)
Common ← POC (poc-app, poc-jira)

Admin ← Export
Admin ← Event (event-client, event-server)

Event (event-client) ← Event (event-server)    ← 父模块内部
Event (event-client) ← poc-app                 ← 运行时通过 Feign 调用 event-server
Event (event-client) ← poc-jira                ← 运行时通过 Feign 调用 event-server
```

### 3.3 服务通信拓扑（运行时）

```
                          OpenFeign (HTTP)
       ┌──────────────────────────────────────────────────┐
       │                    ┌───────────────┐             │
       │     ┌─────────────│ SDLC_Event    │──────────┐  │
       │     │             │    :8081      │          │  │
       │     │             └───────┬───────┘          │  │
       │     │                     │                  │  │
       │     ▼                     ▼                  ▼  │
  ┌─────────┐              ┌──────────┐         ┌─────────┐
  │ poc-app │              │  Oracle  │         │poc-jira │── HTTP ──→ Jira Cloud
  │  :8080  │              │ (共用DB) │         │  :8082  │
  └─────────┘              └──────────┘         └─────────┘
       │                        ▲                     │
       └────────────────────────┘─────────────────────┘
           三者均通过 Feign 互相调用，共用同一个 Oracle
```

- **poc-app** (:8080)：主业务入口，前端调这个。需要校验时调 Event，需要 Jira 操作时调 poc-jira。
- **poc-jira** (:8082)：Jira 集成服务，封装 JiraGateway。挂了或重启不影响主应用。
- **event-server** (:8081)：规则引擎。poc-app 和 poc-jira 都可以调它做校验。
- 三个服务共用同一个 Oracle 数据库（Phase 1-2 期间）。
- 所有跨服务通信走 OpenFeign，不通过 JVM 内方法调用。

### 3.4 为什么不做更多拆分？

Export 和 Admin 暂时不拆成独立进程。原因：

1. Admin 是 Export 和 Event 的共同上游，拆它影响面太大
2. Export 不涉及循环依赖，改造成本低但收益也低（上线后再拆很容易）
3. Jira 必须拆：poc-jira 独立部署后，Jira API 超时或 poc-jira 重启不会影响主应用——这是上线后的韧性保障
4. 3 天时间聚焦于「不改就来不及了」的问题：循环依赖、Jira 韧性、Redis 真实化

## 4. Day 1：创建 Event 微服务 + 解除循环依赖

**目标**：Event 与 POC 编译期完全解耦，5 处 @Lazy 全部移除，Event 成为可独立部署的 Spring Boot 应用。

### 4.1 执行步骤

#### Step 1 — 改造 SDLC_Event 为父模块

修改 `SDLC_Event/pom.xml`，将 packaging 从 `jar` 改为 `pom`：

```xml
<!-- SDLC_Event/pom.xml -->
<packaging>pom</packaging>
<modules>
    <module>event-client</module>
    <module>event-server</module>
</modules>
```

删除原 `SDLC_Event/src` 目录（内容将在后续步骤中分配到子模块）。

#### Step 2 — 创建 event-client 子模块

```xml
<!-- SDLC_Event/event-client/pom.xml -->
<packaging>jar</packaging>
<dependencies>
    <dependency>SDLC_Common</dependency>
    <!-- 不依赖 Admin、不依赖 POC -->
</dependencies>
```

迁入内容：

| 类别 | 目标包路径 | 来源 |
|------|-----------|------|
| 注解 | `com.sdlc.event.api.annotation.ExecutionPhase` | 原 Event |
| 注解 | `com.sdlc.event.api.annotation.ExecutionType` | 原 Event |
| SPI 接口 | `com.sdlc.event.api.spi.Validator` | 原 Event |
| SPI 接口 | `com.sdlc.event.api.spi.Handler` | 原 Event |
| 上下文 DTO | `com.sdlc.event.api.context.*` | 原 Event |
| 共享 Entity | `com.sdlc.event.api.entity.*` | 原 POC（Event 引用的那部分） |
| 共享枚举 | `com.sdlc.event.api.enums.*` | 原 POC（Event 引用的那部分） |
| 共享 VO/DTO | `com.sdlc.event.api.dto.*` | 原 POC（Event 引用的那部分） |
| **Feign 接口**（新增） | `com.sdlc.event.api.feign.EventFeignClient` | **新建** |

##### 如何精准定位要迁出的类型

使用 Copilot Ask（约 1 次额度），提问：

> "在 SDLC_Event 模块中，搜索所有 import com.sdlc.poc 的语句，按 Entity / VO / Enum / Service / DTO 分类列出完整的类名清单。对每个类，判断它是否被 Event 模块的 Validator 或 Handler 直接引用。只输出被直接引用的类。"

如果无法用 Copilot Ask，用命令行：

```bash
grep -rn "import com\.sdlc\.poc\." SDLC_Event/src/ --include="*.java" \
  | grep -v "test" \
  | sed 's/.*import //' | sed 's/;//' | sort -u
```

所得的每个类型，手动确认是否在 Validator/Handler 中被用作方法参数/返回值/字段。满足任一条件即需迁移。

##### Feign 接口定义

```java
// event-client: com.sdlc.event.api.feign.EventFeignClient
@FeignClient(name = "sdlc-event", url = "${event.service.url:http://localhost:8081}")
public interface EventFeignClient {

    @PostMapping("/api/event/validate")
    List<ValidationResult> validate(@RequestBody ValidationRequest request);

    @PostMapping("/api/event/execute")
    ExecutionResult execute(@RequestBody ExecutionRequest request);

    @GetMapping("/api/event/health")
    String health();
}
```

`ValidationRequest` 和 `ExecutionRequest` 是 DTO，封装 type、action、上下文数据——与当前 POC 传给代理 Service 的参数一一对应。

#### Step 3 — 创建 event-server 子模块

```xml
<!-- SDLC_Event/event-server/pom.xml -->
<packaging>jar</packaging>

<parent>spring-boot-starter-parent</parent>

<dependencies>
    <dependency>SDLC_Common</dependency>
    <dependency>SDLC_Event:event-client</dependency>
    <dependency>SDLC_Admin</dependency>  <!-- 保留：原有业务依赖 -->
    <dependency>spring-boot-starter-web</dependency>
</dependencies>
```

迁入内容：
- 原 Event 模块的所有 **Validator 实现类**
- 原 Event 模块的所有 **Handler 实现类**
- 原 Event 模块的所有内部工具类（不被外部引用的）
- **新建启动类** `EventServerApplication.java`
- **新建 REST Controller** 作为 Feign 的入口

##### 启动类

```java
// event-server: com.sdlc.event.server.EventServerApplication
@SpringBootApplication
@ComponentScan(basePackages = "com.sdlc.event")
public class EventServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EventServerApplication.class, args);
    }
}
```

##### REST Controller（Feign 入口）

```java
// event-server: com.sdlc.event.server.controller.EventController
@RestController
@RequestMapping("/api/event")
public class EventController {

    private final EventExecutionService executionService;

    @PostMapping("/validate")
    public List<ValidationResult> validate(@RequestBody ValidationRequest request) {
        return executionService.validate(request);
    }

    @PostMapping("/execute")
    public ExecutionResult execute(@RequestBody ExecutionRequest request) {
        return executionService.execute(request);
    }
}
```

`EventExecutionService` 内部逻辑就是**原来 POC 中的代理 Service**——按 type/action 查找对应的 Validator/Handler，编排执行。这段逻辑从 POC 搬到 event-server。

#### Step 4 — 调整 POC 模块

**pom.xml 变更**：

```xml
<!-- POC/pom.xml -->
<!-- 删除 -->
<dependency>SDLC_Event</dependency>

<!-- 新增 -->
<dependency>SDLC_Event:event-client</dependency>
<dependency>spring-cloud-starter-openfeign</dependency>
```

**代码变更**：

1. 更新所有迁出类型的 import（poc-app/poc-jira 中引用 event-client 的 entity/dto/enum）
2. 删除原来的代理 Service（已搬到 event-server 的 EventExecutionService）
3. 在需要调用 Event 的地方，注入 `EventFeignClient` 替代原来的本地代理 Service
4. 在 `PocApplication` 和 `JiraServerApplication` 分别加 `@EnableFeignClients`

**配置变更**（application.yml）：

```yaml
event:
  service:
    url: http://localhost:8081  # 开发环境；上线前改为环境变量
```

#### Step 5 — 移除 @Lazy

执行 `grep -rn "@Lazy" SDLC_POC/src/ --include="*.java"`，逐个检查：

- 如果该 @Lazy 是用于解决 Event ↔ POC 循环依赖 → **删除**
- 如果该 @Lazy 是其他用途（如懒加载大对象）→ **保留**

删除后启动 event-server（:8081）和 POC 主应用，验证能正常通信和编译。

### 4.2 验证标准

| 检查项 | 命令 | 期望 |
|--------|------|------|
| 全量编译 | `mvn clean compile -pl .` | 通过 |
| Event 不依赖 POC | `grep -r "com.sdlc.poc" SDLC_Event/ --include="*.java"` | 空 |
| poc-app 不依赖原 Event | `grep -r "com.sdlc.event.impl\|com.sdlc.event.service" SDLC_POC/poc-app/ --include="*.java"` | 空 |
| @Lazy 仅剩非循环用途 | 人工审查 | 不超过 3 处 |
| event-server 独立启动 | `mvn spring-boot:run -pl SDLC_Event/event-server` | 正常，端口 8081 |
| Feign 调通 | `curl http://localhost:8081/api/event/health` | 200 OK |

## 5. Day 2：Redis 独立模块 + POC 拆分为 poc-app/poc-jira + Feign 编排

**目标**：Redis 从内存模拟切换到真实 Redis，并独立为 SDLC_Redis 模块；POC 拆分为 poc-app（主业务）和 poc-jira（Jira 集成服务），两者均可独立部署，poc-jira 重启不影响主应用。

### 5.1 创建 SDLC_Redis 模块

#### 模块定位

```
SDLC_Redis（jar，被所有模块引用）
├── 依赖：SDLC_Common
├── 无上游依赖（Common 除外）
└── 提供统一的缓存操作接口
```

#### 迁移内容

| 项目 | 原位置 | 目标位置 |
|------|--------|---------|
| RedisTemplate 配置类 | POC 或 Common（内存模拟的那份） | SDLC_Redis |
| 缓存 Key 枚举/常量 | Common | SDLC_Redis |
| 封装的操作接口（CacheService） | 各 Service 中散落 | SDLC_Redis |

#### 接口设计

```java
// SDLC_Redis: com.sdlc.redis.CacheService
public interface CacheService {
    <T> Optional<T> get(String key, Class<T> type);
    <T> void set(String key, T value, Duration ttl);
    void delete(String key);
    boolean exists(String key);
}
```

#### 配置

```yaml
# application.yml（各引用方配置）
spring:
  redis:
    host: ${REDIS_HOST:localhost}
    port: ${REDIS_PORT:6379}
    timeout: 3000ms
```

**关键**：Redis 配置写在**引用方**的 application.yml 中，SDLC_Redis 模块只提供自动配置类（`@Configuration` + `@ConditionalOnClass`），不硬编码连接信息。

#### 另一个后端影响

这是整个 3 天里**唯一**影响另一个后端的步骤。他需要：

1. 本地安装 Redis（`docker run -d -p 6379:6379 redis:7-alpine`）
2. 拉你的代码后，在自己的 application.yml 加 Redis 配置
3. 确认编译通过

**建议**：Day 2 早上先和他同步，确保他 Redis 就绪后再动手改。

#### 设计变更说明（与初版设计的关键差异）

原设计 POC 是单个应用内部分包（core/jira）。根据用户确认，改为 **POC 父模块 + 两个独立部署的子应用**：

- **poc-app** (:8080)：主业务服务。去掉 Jira 交互后的纯业务逻辑，Issue、Tollgate、Attachment、Tailoring 等 Controller 全在这里。
- **poc-jira** (:8082)：Jira 集成服务。封装 JiraGateway，独立部署。挂了或重启不影响主应用——这是上线后的韧性保障。

**为什么 Jira 必须独立部署？**

Jira Cloud API 是外部依赖，不可控。如果 Jira 调用超时、返回 5xx、或 poc-jira 需要重启更新 Jira 连接配置，这些场景下主应用 poc-app 的业务接口不应该受影响。Jira 操作失败时，poc-app 可以优雅降级（例如提示用户"Jira 同步暂时不可用，请稍后重试"），而不是整个系统卡住。

#### 当前问题（Day 2 开始时）

```
POC/（单应用，port 8080）
├── controller/
│   ├── IssueController.java       ← 业务入口（保留）
```

**Step 1 — 将 POC 改造为父模块**

```java
// poc-jira: com.sdlc.poc.jira.api.feign.JiraFeignClient
@FeignClient(name = "sdlc-poc-jira", url = "${jira.service.url:http://localhost:8082}")
public interface JiraFeignClient {

    @PostMapping("/api/jira/issue")

    @GetMapping("/api/jira/issue/{issueKey}")
```

**Step 5 — 改写 poc-app 中的 Jira 调用点**

之前：
```java
// poc-app: IssueService（旧）
@Autowired
private JiraIssueService jiraIssueService; // 本地 Bean

public Issue createAndSyncToJira(IssueEntity entity) {
    issueDao.insert(entity);
    jiraIssueService.createJiraIssue(entity); // 直接调用，耦合
    return entity;
}
```

之后：
```java
// poc-app: IssueService（新）
@Autowired
private JiraFeignClient jiraFeignClient;      // Feign 远程调用

public Issue createAndSyncToJira(IssueEntity entity) {
    issueDao.insert(entity);
    try {
        JiraIssueRequest req = toJiraRequest(entity);
        JiraIssueResponse resp = jiraFeignClient.createIssue(req);
        entity.setJiraKey(resp.getIssueKey());
        issueDao.update(entity);
    } catch (FeignException e) {
        // Jira 不可用时优雅降级，不阻塞主流程
        log.warn("Jira sync failed for issue {}, will retry later", entity.getId());
    }
    return entity;
}
```

## 7. 各专题深度解答

### 7.1 Oracle → PostgreSQL 迁移策略

**最佳时机**：Phase 2 后期，即功能基本稳定、上线前约 2-4 周。

**为什么不在 Phase 1 做？**
- 数据库迁移是高摩擦操作：要改 SQL 方言、数据类型映射、序列/自增主键、分页语法等
- Phase 1 同时在重构模块，两个高变更操作叠加风险太高
- 系统未上线，没有数据迁移的历史包袱（10 张表，数据量小）

**具体迁移策略（四步走）**：

**Step 1 — SQL 方言适配（先行）**

在 Phase 2 开发过程中，就开始避免 Oracle 特有语法：
- 用 MyBatis 的标准 XML，不写数据库特定 SQL
- 分页统一用 MyBatis PageHelper（自动适配方言）
- 序列改为 MyBatis-Plus 的 `@TableId(type = IdType.AUTO)`，配合 PostgreSQL 的 SERIAL
- 日期函数统一 Java 层处理，不依赖 `SYSDATE`、`TO_DATE` 等

**Step 2 — 引入 Flyway**

```xml
<dependency>
    <groupId>org.flywaydb</groupId>
    <artifactId>flyway-core</artifactId>
</dependency>
```

所有 DDL 用 Flyway 脚本管理（V1__xxx.sql），Oracle 和 PostgreSQL 各一套 baseline。

**Step 3 — 双写灰度切换**

1. 功能测试阶段，新数据同时写 Oracle 和 PostgreSQL
2. 数据一致性校验通过后，读切换到 PostgreSQL
3. 跑一周无问题后，Oracle 下线
4. 这种方式只适用于 10 张表的规模——表多了复杂度爆炸

**Step 4 — 切换**

- 上线前一次性切到 PostgreSQL
- Oracle 保留 1 个月作为回滚保险

### 7.2 微服务拆分前的准备工作

在 Phase 3（正式拆分）开始前，必须确保以下事项就绪：

### 7.3 循环依赖治理详解

#### 当前状态

SDLC_Event 和 SDLC_POC 之间 5 处循环依赖，@Lazy 掩盖。依赖本质：

```
POC → Event：POC 的代理 Service 需要注入 Event 的 Validator/Handler 实现类
Event → POC：Event 的 Validator/Handler 需要引用 POC 的 Entity/VO/Enum
```

#### 解法：依赖倒置 + 接口分离

```
                 event-client
                 （纯接口 + DTO）
                 /            \
                /              \
        event-server        poc-app / poc-jira
   （Validator 实现）     （注入 FeignClient）
```

**核心原理**：把 Event 依赖 POC 的部分（Entity/DTO/Enum）提升到一个 POC 和 Event 都可达的中间层（event-client），同时把 POC 依赖 Event 的部分（注入 Validator）从编译期依赖改为运行时 HTTP 调用。

#### @Lazy 的去留

**全部移除**。理由：

1. @Lazy 解决的是 Spring 容器初始化时的 Bean 创建顺序问题，不解决架构问题
2. 上线后 @Lazy 可能在并发场景下暴露时序问题
3. 新架构中 poc-app/poc-jira 和 event-server 在不同 JVM 中运行，根本没有「Bean 创建顺序」的概念

移除验证：`grep -rn "@Lazy" --include="*.java"` 结果为空（业务需要的除外）。

#### 治理原则（写入团队规范）

```java
// ❌ 永远不允许
// A 模块的类 import B 模块的类，且 B 模块的类 import A 模块的类

// ✅ 解耦模式
// 1. 共享类型提升到公共层
// 2. 接口定义在低层，实现在高层
// 3. 跨模块调用走 Feign（运行时）而非 import（编译期）
```

这条规则由 ArchUnit 自动检查（见 6.2 节），违规则 CI 失败。

### 7.5 为何不使用 Nacos

**结论：完全合理，不需要 Nacos。**

理由：

| 维度 | Nacos 解决的问题 | 你们的替代方案 |
|------|-----------------|--------------|
| 服务注册与发现 | 需要 | AWS ECS Service Discovery / K8s Service / DNS |
| 配置中心 | 需要 | AWS AppConfig / Parameter Store / Spring Cloud Config |
| 动态路由 | 不需要（在线协同平台不需要复杂的灰度路由） | — |
| 流量管理 | 不需要（不是高并发互联网产品） | — |

**你们不需要 Nacos 的核心原因**：

1. AWS ECS Service Discovery 已经解决了服务发现（DNS 级别，零额外维护）
2. 配置管理用 AWS AppConfig 或 Parameter Store 足够，且与 AWS 部署体系一致
3. 万级用户的企业内部系统不需要 Nacos 的流量治理能力（权重路由、金丝雀等）
4. 部署在 AWS 上，引入 Nacos 意味着要么用 Nacos Cloud（托管，额外费用），要么自建 Nacos 集群（运维负担）。两种都不如用 AWS 原生服务经济

**类比**：Nacos 适合阿里云生态或自建机房的大型微服务集群（几十上百个服务）。你们的规模（上线后 3-5 个服务，万级用户），用 AWS 原生方案够了。

### 7.6 OpenFeign + AWS Service Discovery 可行性

**结论：完全足够，这是 AWS 生态的标准搭配。**

#### 架构示意

```
┌──────────┐  Feign 调用   ┌──────────────┐  DNS 解析   ┌──────────┐
│ poc-app  │ ────────────→ │ OpenFeign     │ ──────────→ │ ECS      │
│  :8080   │               │ + Ribbon/LB  │            │ Service  │
└──────────┘               └──────────────┘            │ Discovery│
                                                       └────┬─────┘
                                                            │
                                                            ▼
                                                    ┌──────────────┐
                                                    │ Event Server │
                                                    │   :8081      │
                                                    └──────────────┘
```

#### 具体配置

```yaml
# 当前（Phase 1：单机开发）
event:
  service:
    url: http://localhost:8081

# 上线前（Phase 2 后期：AWS 部署）
spring:
  cloud:
    openfeign:
      client:
        config:
          sdlc-event:
            connectTimeout: 5000
            readTimeout: 10000
```

```java
// Feign 接口：开发时用 URL，AWS 上切换为 Service Discovery
@FeignClient(
    name = "sdlc-event",
    url = "${event.service.url}",  // 开发环境
    fallbackFactory = EventFallbackFactory.class
)
public interface EventFeignClient { /* ... */ }
```

AWS ECS Service Discovery 基于 Route 53，自动为每个 ECS Service 注册 DNS 记录（如 `sdlc-event.local`）。OpenFeign 只需配置 service name，底层走 DNS 解析 → 负载均衡。

#### 为什么不需要额外的服务发现组件（Consul/Eureka）

- AWS ECS Service Discovery 提供的是 DNS A 记录，OpenFeign + Spring Cloud LoadBalancer 默认支持 DNS 解析
- 不需要维护额外的服务发现集群
- 和 ECS 滚动更新/健康检查天然集成

### 7.7 未来服务边界划分

#### 划分原则

对你们这个系统，用 **业务能力** + **数据所有权** 两个维度划分：

| 服务 | 业务能力 | 拥有的数据（未来独立 DB 时） |
|------|---------|---------------------------|
| **POC Core** | Issue 管理、Tollgate 审批、附件管理、字段映射 | Issue 表族、Tollgate 表族、Attachment 表 |
| **Event** | 规则校验、事件处理、流程编排 | Event 配置表、校验规则表、执行日志表 |
| **Admin** | 系统管理、用户权限、项目配置 | 用户表、权限表、项目表、配置表 |
| **Export** | 报表导出、数据查询、批量操作 | 导出任务表、导出历史表（主要读已有数据） |
| **Jira Gateway** | Jira 集成、同步调度 | Jira 映射表、同步队列表、同步日志表 |

**注意**：Phase 1 中 POC Core 已拆为 poc-app，Jira Gateway 已拆为 poc-jira 独立服务。Phase 3 是将 Admin 和 Export 从 jar 依赖拆成独立进程。

#### 不建议再做更细拆分

不要把 Issue 和 Tollgate 拆成两个服务。它们有大量交叉查询（一个 Tollgate 审批关联多个 Issue），当前共用 DB 看不出问题，未来独立 DB 后跨服务 join 会很难处理。属于同一个限界上下文（Bounded Context）的业务，留在同一个服务里。

#### 建议的拆分顺序

1. **Event**（Phase 1 已完成）— 最简单，价值最大
2. **Jira Gateway（poc-jira）**（Phase 1 已完成）— 独立性强，保障主应用韧性
3. **Export** — 只读为主，风险可控
4. **Admin** — 最后拆，因为它被多个服务依赖，需要先让其它服务有自己的权限判断方式

### 7.8 现在做 vs 预留扩展点

#### 现在做（Phase 1，本次 3 天）

| 事项 | 理由 |
|------|------|
| 解除 Event ↔ POC 循环依赖 | 不改 = 每次改代码都互相牵连 |
| Event 独立为微服务 | 规则引擎天然独立；花费小、收益大 |
| poc-jira 独立为微服务 | Jira 外部依赖不可控，独立部署保障主应用韧性 |
| Redis 模块化 + 切换真实 Redis | 内存模拟不可上线 |
| POC 拆分为 poc-app + poc-jira | 不改 = Jira 代码散落，未来切同步模式要全量翻改 |
| CI/CD 流水线 | 不改 = 每合并一次烂代码就多还一次债 |
| ArchUnit 架构测试 | 不改 = 循环依赖和越界引用必然再出现 |
| JiraGateway 接口 + 策略模式 | 只加接口，不写实现（Direct 照搬旧代码） |

#### 预留扩展点（只定义接口，不实现）

| 事项 | 预留方式 | 何时实现 |
|------|---------|---------|
| SyncDbJiraGateway | `@ConditionalOnProperty` + 接口 | 产品确认切换同步模式时 |
| Jira 同步队列表 | DDL 脚本就绪，不执行 | 上线后视数据量决定是否切换 |
| 服务间熔断/降级 | Feign fallbackFactory 接口占位 | 微服务数量 ≥ 3 且线上有跨服务调用失败时 |
| 分布式链路追踪 | Sleuth 依赖引入 + log pattern 占位 | Phase 3 拆分前 |
| 配置中心 | 环境变量替代硬编码（已做）| Phase 3 接入 AWS Parameter Store |
| PostgreSQL 迁移 | Flyway 脚本 + 方言适配（开发期顺手做）| Phase 2 后期执行 |

#### 明确不做（YAGNI）

| 事项 | 理由 |
|------|------|
| 消息队列（Kafka/RabbitMQ） | 没有异步解耦需求；共用 DB 暂不需要 |
| 分布式事务框架（Seata） | 共用 DB，Phase 1-2 无分布式事务 |
| Elasticsearch 搜索 | 当前数据量和查询复杂度不需要 |
| API 网关（Spring Cloud Gateway） | 服务数 < 5 时网关边际效益低；可后续加 |
| 服务网格（Istio/Linkerd） | 服务数 < 10 时运维成本远超收益 |

### 7.9 AI 工具最大化利用策略

#### 三种 AI 的分工

| AI | 特性 | 最佳用途 | 禁止用途 |
|----|------|---------|---------|
| **ChatGPT 5.5** | 无限、不能读源码、擅架构 | 设计审查、代码片段生成、配置生成、文档生成 | 不能做需要读实际代码的任务 |
| **Copilot Ask** | 10 次、能读当前文件/模块 | 跨模块依赖分析、import 扫描、重构影响范围评估 | 用完 10 次就别用了 |
| **Copilot Agent** | 1 次、能读整个项目 | 最终架构审查：全项目依赖关系图 + 异常报告 | 不要在前面步骤用掉 |

#### 使用原则

1. **ChatGPT 是主力**：设计、配置、模板代码、用例分析——全给它
2. **Copilot Ask 是精准手术刀**：只在需要「读源码 + 分析」时用，每次用完问自己「这次得到的信息是否 ChatGPT 也能做？」
3. **Copilot Agent 是最终安全网**：留到最后，让它做全项目扫描。前面的步骤做得再好，总会有遗漏——Agent 是兜底

#### 每个 AI 的推荐提问模板

**ChatGPT 典型提问**：

> 你是一名 Java 架构师，精通 Spring Boot 多模块项目和 Spring Cloud 微服务。背景：我在重构一个企业内部协同平台，模块结构如下 [贴模块依赖]。请帮我审查以下设计方案 [贴设计]，重点看：1. 依赖方向是否有遗漏的回环；2. 模块职责是否清晰；3. 是否有可简化的地方。给出具体建议，不要通用理论。

**Copilot Ask 典型提问**：

> 在当前模块中，搜索所有 import com.sdlc.event 的语句。按文件路径分类列出，标注每个文件 import 了 Event 模块的哪些类，以及这些类的用途（接口/实现/注解/实体）。输出 Markdown 表格。

**Copilot Agent 典型提问**：

> 对整个项目做架构分析。输出：1. 模块依赖有向图（Mermaid 格式）；2. 循环依赖检测（包括传递性的）；3. 违反模块边界规则的代码位置（例如 poc-app 引用 poc-jira 内部实现）；4. 未被共享层封装的跨模块 import。每个问题标注文件路径和行号。

#### 用完之后

重构结束后，你还有大约 5 次 Copilot Ask 额度。建议保留给 Phase 2 开发中的**复杂 SQL 优化**或**MyBatis 映射调试**——这是 ChatGPT 帮不了（它看不到你的实际 SQL 和表结构）、但又很费人工的事。
